# Design: Lingo — Self-Hosted Company Glossary

Generated by /office-hours on 2026-03-25
Status: DRAFT
Mode: Builder

## Problem Statement

Every company accumulates a dense vocabulary of internal acronyms, service names, project
codenames, and org-specific phrases. New hires spend weeks confused. Existing employees
waste time answering the same "what does X mean?" questions. Existing solutions (Confluence,
Notion, Google Docs) fail because they are **not in the flow** — you have to leave where you
encountered the term, go find the doc, hope it's up to date, and come back.

Lingo makes the glossary live where you work: Slack, CLI, and AI agents.

## What Makes This Cool

- **In-flow lookup:** Type `/lingo define BART` in Slack and get the answer without leaving the conversation.
- **AI-native:** An MCP endpoint means any AI agent (Claude, Cursor, Zed) can automatically
  resolve internal terms it encounters in code, docs, or tickets.
- **Community-legitimized:** Terms earn "Official" status through votes — no gatekeeper bottleneck,
  but editors can fast-track important terms.
- **Self-hostable in under 5 minutes:** Docker-compose, plug in your IdP, you're done.

## Constraints

- Must support Google Workspace, Okta, or any SAML-compatible IdP for auth
- Must be self-hostable (Docker/docker-compose); no SaaS dependency
- Accessible to non-developers (web UI), developers (CLI), and AI agents (MCP/REST API)
- Slack integration must work with Slack's existing OAuth flow (requires workspace admin approval)
- **Note on setup time:** OIDC/SSO configuration requires IdP admin access and app registration, which may take hours to days depending on the organization. The "5-minute" promise applies once SSO credentials are in hand. A `LINGO_DEV_MODE=true` flag enables a local password auth bypass for evaluation/development; this mode is explicitly disabled in production and shows a warning banner in the web UI.

## Premises

1. **Accessibility-in-the-flow is the core insight.** The right place to look up a term
   is where you encounter it — not a separate wiki tab.

2. **API-first architecture.** One canonical data store with a REST + MCP API.
   Slack bot, web UI, and CLI are all thin clients over this API.

3. **"Official" status has two paths:**
   - Community path: reaching a vote threshold (e.g., 3+ thumbs = Community, 10+ = Official)
   - Editor path: designated editors can mark terms Official directly
   Both paths result in the same "Official" state. Editors are assigned by admins.

4. **Self-hostable means Docker/docker-compose, <5 min deploy.**
   Minimal dependencies. No managed cloud required.

5. **Auth via SSO only.** Google Workspace, Okta, or generic SAML/OIDC.
   No username/password auth. If you're not in the IdP, you can't access it.

## Approaches Considered

### Approach A: API + CLI Core Only
Minimal scope. REST + MCP API with a CLI wrapper. Slack bot and web UI deferred.
- Effort: S | Risk: Low
- Ships fastest, but leaves out the highest-adoption surface (Slack)

### Approach B: Full Stack — API + CLI + Slack Bot + Web UI (CHOSEN)
Same API core, but ships Slack bot (/lingo define, /lingo add, /lingo vote), web UI
(browse, search, vote, add terms), and CLI. Docker-compose bundles everything.
- Effort: M | Risk: Medium
- Full adoption from day one; every role can participate

### Approach C: MCP-First
MCP as the primary interface, CLI as thin wrapper, Slack calls MCP.
- Effort: M | Risk: Low-Medium
- Forward-looking but over-indexes on AI agents at the expense of human UX

## Recommended Approach

**Approach B — Full Stack.**

The wow moment is both Slack AND AI agent integration. Approach B delivers both.
The MCP endpoint is an additional route on the same REST API — it adds minimal
complexity but completes the vision. The three surfaces (web, Slack, CLI) share
the same data layer and auth.

**Tech stack recommendation:**
- **Backend:** Go — single binary, no runtime dependency, ideal for self-hosted Docker
- **Database:** Postgres (pgx driver) — native FTS, proper concurrency, no dual-driver complexity
- **ORM/query layer:** sqlc — type-safe Go codegen from SQL schema; schema-first migrations
- **Auth:** OIDC/SAML middleware — supports Google Workspace, Okta, generic SAML
- **Slack:** slack-go/slack in Socket Mode — no public URL required, works behind firewalls
- **MCP:** `/mcp` endpoint (custom HTTP handler) — auth via API tokens (Bearer)
- **CLI:** `lingo` — client CLI binary; uses cobra. If a server binary is needed, it ships as `lingo-server`.
- **Frontend:** React + Vite — compiled to `/static/`, served by Go; Tailwind CSS + shadcn/ui
- **Deploy:** Docker + docker-compose (lingo-server + postgres); Helm chart for Kubernetes

**Status model:**
```
Suggested (auto-discovered from Slack, not yet human-reviewed)
  ↕ Pending   (promoted by editor, or user-created; 0 to LINGO_COMMUNITY_THRESHOLD-1 votes)
    ↕ Community  (LINGO_COMMUNITY_THRESHOLD+ votes OR editor approval)
      ↕ Official   (LINGO_OFFICIAL_THRESHOLD+ votes OR editor marks official)
```
Thresholds are configurable via environment variables (defaults: community=3, official=10).
Tune per org size — a 10-person team may use 2/5; a large org may use 5/25.
Staleness checks exclude `suggested` terms.
**All status transitions are reversible by editors.** Demotions require a `change_note` and create a `Term_History` record. Votes are not reversed on demotion (vote count is preserved).

**Voting mechanics:**
- Any Member can upvote a term (one vote per user per term, enforced at DB level).
- `POST /api/v1/terms/:id/dispute` — notifies term owner via Slack DM.

**Role model:**
- **Member** — can add terms, vote, browse, claim ownership of unowned terms
- **Editor** — can mark terms Official, edit any term (including suggested), delete terms, manage relationships, promote/dismiss suggestions
- **Admin** — can assign Editor roles, manage SSO config, create/revoke API tokens

## Data Model

All IDs are UUID v4.

```
Term {
  id                uuid PK
  name              text NOT NULL
  full_name         text
  definition        text NOT NULL CHECK (length(definition) <= 2000)
  category          text
  status            text NOT NULL  -- pending | community | official | suggested
  source            text NOT NULL  -- 'user' | 'slack_discovery'
  source_channel_id text            -- nullable; slack channel ID with highest occurrence (slack_discovery only)
  occurrences_count int             -- only set for suggested terms
  owner_id          uuid FK → User  -- nullable
  owned_at          timestamptz
  is_stale          boolean NOT NULL DEFAULT false
  last_confirmed_at timestamptz     -- defaults to created_at; resets on /confirm
  version           int NOT NULL DEFAULT 1  -- optimistic locking; PUT requires matching version
  created_by        uuid FK → User
  created_at        timestamptz
  updated_at        timestamptz
  search_vector     tsvector GENERATED ALWAYS AS (
    to_tsvector('english',
      coalesce(name,'') || ' ' || coalesce(full_name,'') || ' ' || coalesce(definition,'')
    )
  ) STORED
}
INDEX terms_search_vector_idx ON terms USING GIN (search_vector)

Term_Relationship {
  id               uuid PK
  term_id          uuid FK → Term NOT NULL
  related_term_id  uuid FK → Term NOT NULL
  relationship_type text NOT NULL  -- depends_on | supersedes | related_to
  created_by       uuid FK → User  -- editor+
  created_at       timestamptz
}

Term_History {
  id          uuid PK
  term_id     uuid FK → Term NOT NULL
  definition  text
  full_name   text
  category    text
  owner_id    uuid
  status      text
  changed_by  uuid FK → User NOT NULL
  changed_at  timestamptz NOT NULL
  change_note text  -- optional, max 280 chars; from PUT body
}

Vote {
  term_id    uuid FK → Term
  user_id    uuid FK → User
  created_at timestamptz
  PRIMARY KEY (term_id, user_id)  -- enforces one vote per user per term
}

User {
  id            uuid PK
  email         text NOT NULL UNIQUE
  display_name  text
  slack_user_id text  -- nullable; populated on first Slack interaction
  role          text NOT NULL  -- member | editor | admin
  last_login_at timestamptz
  is_active     boolean NOT NULL DEFAULT true
  created_at    timestamptz
}

Token {
  id           uuid PK
  user_id      uuid FK → User
  name         text        -- human label, e.g. "cursor-agent"
  token_hash   text        -- sha256(raw_token); raw token shown once at creation
  scopes       text[]      -- ["read"] for MCP tokens
  created_at   timestamptz
  last_used_at timestamptz
  revoked_at   timestamptz  -- null = active
}
-- Token generation: crypto/rand 32 bytes → base64url
-- Token storage: sha256(token) — fast lookup, no bcrypt latency on MCP calls

Job {
  id            uuid PK
  job_type      text NOT NULL  -- 'discovery' | 'staleness'
  status        text NOT NULL  -- pending | running | completed | failed
  progress_json jsonb           -- { channels_done, channels_total, cursor }
  started_at    timestamptz
  completed_at  timestamptz
  error         text
}
```

## API Routes

All REST routes are prefixed `/api/v1/`. The MCP endpoint follows the MCP spec and is unversioned.

```
-- Terms
GET    /api/v1/terms                           — list/search (?q=, ?status=, ?category=)
POST   /api/v1/terms                           — add a term (member+)
GET    /api/v1/terms/:id                       — get term (includes relationships[], linked_terms[])
PUT    /api/v1/terms/:id                       — update term; requires `version` field (optimistic lock);
                                                 triggers Term_History snapshot;
                                                 member for own user-created terms; editor+ for others + all suggested
DELETE /api/v1/terms/:id                       — delete (editor+)
POST   /api/v1/terms/:id/vote                  — upvote (member+; transactional; deduped at DB level)
POST   /api/v1/terms/:id/dispute               — dispute/downvote (member+; notifies owner via Slack DM)
POST   /api/v1/terms/:id/official              — mark official (editor+)
POST   /api/v1/terms/:id/confirm               — confirm not stale (owner or editor+; resets last_confirmed_at)
POST   /api/v1/terms/:id/claim                 — claim ownership (member+ if owner_id is null; editor+ to override)
GET    /api/v1/terms/:id/history               — edit history
POST   /api/v1/terms/:id/revert/:history_id    — revert to snapshot (editor+)
POST   /api/v1/terms/:id/relationships         — add relationship (editor+)
DELETE /api/v1/terms/:id/relationships/:rel_id — delete relationship (editor+)
POST   /api/v1/terms/:id/promote               — suggested → pending (editor+)
POST   /api/v1/terms/:id/dismiss               — discard suggestion (editor+)

-- Export
GET    /api/v1/export                          — export Official terms as markdown
                                                 ?status=official&format=markdown
                                                 ?offset= for pagination (no hard cap — paginate properly)

-- Users
GET    /api/v1/users                           — list users (admin only)
PATCH  /api/v1/users/:id/role                  — set role: member|editor|admin (admin only)

-- API Tokens (for MCP auth)
GET    /api/v1/tokens                          — list tokens for current user
POST   /api/v1/tokens                          — create token (admin only)
DELETE /api/v1/tokens/:id                      — revoke token (admin only)

-- Admin operational surface
GET    /api/v1/admin/stats                     — glossary stats (admin only)
GET    /api/v1/admin/jobs                      — list background jobs (admin only)
POST   /api/v1/admin/jobs/:type/run            — trigger job manually: discovery | staleness (admin only)

-- MCP
GET    /mcp                                    — MCP endpoint for AI agents (Bearer token auth)
                                                 tools: get_term, search_terms, list_terms
```

## MCP Tool Signatures

```
get_term(name: string) → {
  id, name, full_name, definition, category, status,
  owner: { display_name } | null,
  relationships: [{ type, related_term: { name, definition } }],
  linked_terms: [{ id, name }]

search_terms(query: string, status?: string, limit?: number) → {
  terms: [{ id, name, full_name, definition, status }]
}

list_terms(category?: string, status?: string, limit?: number, offset?: number) → {
  terms: [{ id, name, full_name, definition, status }],
  total: number
}
```

## Background Jobs

**LingoDiscoveryJob** (`job_type: 'discovery'`)
- Scans 90 days of Slack history per invited channel (public + private only; skips DMs)
- Rate-limited to 45 req/min (Slack Tier 3 limit)
- Async, incremental, restartable — progress stored in `Job.progress_json`
- Detection: regex `/\b[A-Z]{2,6}\b/g` on message text; surfaces term if `occurrences_count >= 3`
- Deduplication: if a term with the same name (case-insensitive) already exists at any status, updates `occurrences_count` only — no duplicate `suggested` term created
- Resulting terms have `status = suggested`, `source = slack_discovery`
- **Source channel tracking:** Store the channel with highest occurrence count as `source_channel_id` on suggested terms (add field to Term: `source_channel_id text` — nullable, populated only for slack_discovery terms)
- **Promotion notification:** When an editor promotes a `suggested` term to `pending` or higher, post a channel message to `source_channel_id`: "📚 Lingo: [TERM] has been added to the glossary! [definition preview] — Thanks to this channel for surfacing it. [View in Lingo]" (only if `source_channel_id` is set)

**LingoStalenessJob** (`job_type: 'staleness'`)
- Runs weekly (cron)
- Checks only `pending | community | official` terms (not `suggested`)
- Flags term as `is_stale = true` if `now() - last_confirmed_at > LINGO_STALE_THRESHOLD_DAYS` (default 180)
- Sends Slack DM to owner with Confirm/Update buttons
- Unowned stale terms go to admin "Needs Review" queue
- Admin notified when owner's `last_login_at > 90 days` (deprovisioning detection)

### Frontend Stack

- **Framework:** React + Vite — compiled to `/static/`, served by Go as static files
- **CSS:** Tailwind CSS — token values map to `tailwind.config.js` CSS variables
- **Components:** shadcn/ui (Radix primitives) — Button, Badge, Dialog, Command (search), Popover

### No-card Rule

Cards (bordered boxes) are forbidden in the term list. They appear only in:
- Setup wizard steps
- Error/empty state illustrations
- The slide-in detail panel (which is a surface, not a card)

## Resolved Design Decisions

These were previously open questions — now resolved:

1. **Fuzzy "Did you mean?" threshold:** Enabled. Edit distance ≤ 2. Not user-configurable in v1 (fixed at server level via `LINGO_FUZZY_THRESHOLD=2`).

2. **Slack `/lingo add` flow:** Opens a Slack modal (not inline text parsing). Command `/lingo add [TERM]` pre-fills Term field. Modal fields: Term, Full Name, Definition (multi-line), Category. Submitted to `POST /api/v1/terms`.

3. **Status reversibility:** All status transitions are reversible by editors. `PUT /api/v1/terms/:id` with `status` field changes to any status. Every status change creates a `Term_History` record; `change_note` is required when demoting (status going down). The status machine is not enforced as one-directional.

4. **Filter logic:** All filters use AND logic. `status=Official&category=Engineering` returns only Official Engineering terms. Multiple selections within the same filter group (e.g., two categories) are OR'd within the group, AND'd with other groups.

5. **Staleness admin queue:** Accessible via Status filter in main UI (filter `is_stale=true`). Admin nav bar link "Needs Review (N)" surfaces this filtered view directly.

## First-Run Setup

On first run with an empty database, Lingo prints a one-time bootstrap token to stdout:

```
================================================
LINGO SETUP: Visit http://localhost:3000/setup
Bootstrap token: lingo-setup-a3f9b2c1d4e5
Expires in 1 hour. This message won't appear again.
================================================
```

The `/setup` route:
- Only accessible if DB has 0 users
- Accepts bootstrap token (expires 1 hour, single-use)
- Walks through: OIDC issuer URL + client_id/secret → test SSO connection → creates admin user
- Permanently blocked after first admin is created
